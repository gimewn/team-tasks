# write-docs

프로젝트 문서 5개를 **의존 순서대로** 작성한다. 나중 문서(아키텍처·DB·API)는
앞 문서(페르소나·유저시나리오)에서 도출되므로, 순서를 지키는 것이 일관성의 전부다.

## 1단계: 현황 파악

`docs/` 폴더를 확인한다.

- 폴더가 없으면 생성한다.
- 아래 5개 파일 중 **없는 것만** 만든다. 이미 있는 파일은 사용자가 "업데이트해줘"라고 하지 않는 한 건드리지 않는다.
- 있는 파일은 다음 문서를 쓸 때 맥락으로 읽는다.

| 순서 | 파일 | 한 줄 역할 |
|------|------|-----------|
| 1 | `docs/personas.md` | 누가 쓰는가 — 모든 문서의 기준점 |
| 2 | `docs/user-stories.md` | 어떻게 쓰는가 — 기능 범위 도출 |
| 3 | `docs/architecture.md` | 어떤 스택으로 만드는가 |
| 4 | `docs/db.md` | 어떤 데이터를 저장하는가 |
| 5 | `docs/api.md` | 어떤 인터페이스를 제공하는가 |

## 2단계: 정보 수집

각 문서를 쓰기 전에 필요한 정보를 모은다.

**앞 단계 문서가 있으면** 먼저 읽는다 — 페르소나 이름, 스택, 테이블 명세 등을
그대로 이어받아야 일관성이 유지된다.

**정보가 부족하면** 핵심 질문 1~2개만 한다. 추측으로 만들어도 되는 수준이면
그냥 만들고 나중에 고치도록 안내한다.

- **페르소나**: 앱을 쓰는 사람의 역할과 기술 수준
- **유저시나리오**: 하루 일과 중 앱을 어떻게 쓰는지
- **아키텍처**: `package.json`·`src/` 구조가 있으면 스택을 추론한다. 없으면 묻는다.
- **DB**: 유저시나리오에서 필요한 데이터 엔티티를 도출한다. 기존 코드에 테이블이 있으면 반영한다.
- **API**: DB 스키마 + 유저시나리오 기반으로 엔드포인트를 도출한다.

## 3단계: 문서 작성

순서대로 작성하되, 각 문서의 형식을 아래 템플릿대로 맞춘다.

---

### personas.md 형식

```markdown
# 페르소나

## [역할명]
- **역할**: …
- **기술 수준**: 상 / 중 / 하
- **주요 목표**: …
- **주요 시나리오**:
  1. …
  2. …
  3. …

(페르소나 2, 3 …)

---

## MVP 헌장

**포함 범위**: …
**제외 범위**: …
```

---

### user-stories.md 형식

페르소나별로 하루 일과를 시간 흐름으로 쓴다. 마지막에 미결 결정 사항을 모은다.

```markdown
# 유저 시나리오: [페르소나 이름]

## 하루 일과
- **[시간]**: [행동 및 목적]
- **[시간]**: …

---

## 미결 결정 사항
1. …
2. …
```

---

### architecture.md 형식

```markdown
# 아키텍처

## 구성도

[ 컴포넌트 A ] ──동작──▶ [ 컴포넌트 B ] ──동작──▶ [ 컴포넌트 C ]
                                  │
                               배포/호스팅
                                  │
                           [ 인프라 ]

## 스택

| 레이어 | 기술 | 비고 |
|--------|------|------|
| 프론트엔드 | … | … |
| API | … | … |
| DB / Auth | … | … |
| 인프라 | … | … |
```

---

### db.md 형식

```markdown
# 데이터베이스 설계

## [테이블명]

```sql
create table [name] (
  id         uuid        primary key default gen_random_uuid(),
  …
  created_at timestamptz not null default now()
);
```

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | uuid | 기본 키 |
| … | … | … |
```

---

### api.md 형식

```markdown
# API

## 인증
…

## 엔드포인트

| METHOD | PATH | 설명 | 인증 |
|--------|------|------|------|
| GET | /api/… | … | 필요 / 불필요 |
…
```

## 4단계: 일관성 체크

모든 문서를 쓴 뒤 다음을 확인한다.

- `user-stories.md`에 등장하는 페르소나가 `personas.md`에 모두 정의되어 있는가?
- `db.md`의 테이블이 유저시나리오에서 도출된 엔티티를 커버하는가?
- `api.md`의 엔드포인트가 `db.md` 테이블과 대응하는가?
- `architecture.md`의 스택이 실제 코드베이스와 일치하는가?

불일치가 있으면 파일에 `> ⚠ 불일치: …` 주석을 달고 사용자에게 알린다.

## 5단계: 완료 보고

```
완료:
  ✓ docs/personas.md       (신규)
  ✓ docs/user-stories.md   (신규)
  - docs/architecture.md   (이미 존재, 건너뜀)
  ✓ docs/db.md             (신규)
  ✓ docs/api.md            (신규)

주의: [일관성 이슈가 있으면 여기에]
```
